# GDELT Global Misinformation Analysis

Event-level analysis of misinformation and disinformation as a global, data-driven phenomenon — using real, directly-extracted news event data from GDELT rather than a pre-made or platform-specific dataset. This project downloads records from GDELT's live Global Knowledge Graph (GKG) archive, filters them on documented misinformation-related theme codes, cleans them into an analytical dataset, and analyzes them to answer three research questions.

**Every record in this repository is traceable back to a real GDELT source file.** `extraction_manifest.csv` logs the exact URL and timestamp queried for every single day covered, so the full raw→clean→analysis chain is independently verifiable end to end.

## Research Questions

1. What types of misinformation themes appear in global news reporting?
2. Where are these events geographically concentrated?
3. Under what political, economic, or public health conditions does misinformation activity increase?

## Methods

| Question | Method | Tools |
|---|---|---|
| What themes appear? | Full-text scraping + spaCy preprocessing + LDA topic modelling | `requests`, `beautifulsoup4`, `spacy`, `gensim` |
| Where do events cluster? | Spatial analytics — DBSCAN, Moran's I, KDE | `scikit-learn`, `geopandas`, `libpysal`, `esda` |
| What drives event spikes? | Count-based regression — Poisson, Negative Binomial | `statsmodels` |

## Data Provenance

All data is extracted directly from GDELT's public GKG v2.1 archive (`data.gdeltproject.org/gdeltv2/`) by `Extraction.py`. No pre-made or third-party dataset is used at any stage.

- **Source:** real GKG files, one fixed daily timestamp per day across the chosen window (documented, reproducible sampling — see Pipeline Overview below)
- **Filter criteria — real, verified GKG theme codes only:**
  `INFO_HOAX`, `INFO_RUMOR`, `MEDIA_CENSORSHIP`, `INTERNET_CENSORSHIP`, `INTERNET_BLACKOUT`, `CYBER_ATTACK`, `SURVEILLANCE`, `HATE_SPEECH`, `FREESPEECH`, `HEALTH_VACCINATION`, `HEALTH_PANDEMIC`
  Each code was checked against GDELT's own [GKG Theme master list](http://data.gdeltproject.org/documentation/GKG-MASTER-THEMELIST.TXT) and codebook before being added — none were assumed or invented.
- **Raw extraction:** `raw_gkg_extraction.csv` holds every matching row exactly as returned by GDELT, with no transformation.
- **Reproducibility record:** `extraction_manifest.csv` lists every source URL queried, the date/timestamp requested, and how many rows it contributed — including days that returned zero matches.

The analytical dataset (`gdelt_cleaned_dataset.csv`) is derived only from this raw extraction, via `Clean_dataset.py`:

- **Identifiers / time:** `Record_ID`, `Date`, `Year`, `Month`, `Year_Month`
- **Geography:** `Country`, `Location_Type`, `Latitude`, `Longitude` — parsed from GDELT's own `V2Locations` field (country names are GDELT's own text, not a separately maintained lookup table)
- **Content:** `Themes` (semicolon-separated real GKG theme codes), `Theme_Count`
- **Tone:** `Tone_Overall`, `Tone_Positive`, `Tone_Negative`, `Tone_Polarity`, `Tone_Activity`, `Tone_SelfGroup` — parsed from GDELT's own `V2Tone` field
- **Source:** `Source` (domain), `Source_URL` (original article URL)

The notebook additionally derives `Political_Context`, `Economic_Context`, and `Health_Context` flags from a second set of theme codes (`GENERAL_GOVERNMENT`, `EPU_POLICY`, `ECON_INFLATION`, `HEALTH_PANDEMIC`, etc.) — also individually verified against the official theme list and/or observed directly in this project's own raw extraction. See the comment block above that cell in the notebook before extending this list.

## Getting Started

### 1. Install dependencies

```bash
pip install -r requirements.txt
```

`geopandas`/`libpysal`/`esda` can be finicky with plain pip on Windows; if you hit build errors, install those specific packages via conda-forge instead:

```bash
conda install -c conda-forge geopandas libpysal esda -y
pip install -r requirements.txt
```

Or, from inside Jupyter itself, run the notebook's second cell (`%pip install ...`), then restart the kernel.

### 2. Run the extraction (Stage 1)

Test on a short date range first:

```bash
python Extraction.py --start 2023-06-01 --end 2023-06-07 --out test_raw.csv --manifest test_manifest.csv
```

Confirm `test_raw.csv` contains real rows and `test_manifest.csv` lists real source URLs. Then run the full window:

```bash
python Extraction.py --start 2022-01-01 --end 2024-01-01 --out raw_gkg_extraction.csv --manifest extraction_manifest.csv
```

This queries one file per day at a fixed time (default 12:15:00 UTC — see `--daily-time`), a documented sampling decision made because GDELT publishes a new file every 15 minutes (~70,000 files across a 2-year window). State this scope limitation explicitly in your methodology.

**This is a long-running job (roughly 30 minutes to a few hours for the full 2022–2024 window).** Keep the machine from sleeping, and don't close the terminal/notebook while it runs.

**If it gets interrupted,** re-run the exact same command with `--resume` added. It reads `extraction_manifest.csv`, skips every day already completed, and appends only what's missing — no lost progress, no duplicates:

```bash
python Extraction.py --start 2022-01-01 --end 2024-01-01 --out raw_gkg_extraction.csv --manifest extraction_manifest.csv --resume
```

**If `raw_gkg_extraction.csv` and `extraction_manifest.csv` ever fall out of sync** (e.g. a separate test run overwrote the manifest but not the raw file — check with `Repair_manifest.py`'s output), rebuild the manifest from what's actually in the raw file before resuming:

```bash
python Repair_manifest.py --raw raw_gkg_extraction.csv --manifest extraction_manifest.csv
```
This is safe to run any time and won't touch days that are already correctly recorded.

### 3. Clean the raw extraction (Stage 2)

```bash
python Clean_dataset.py --raw raw_gkg_extraction.csv --out gdelt_cleaned_dataset.csv
```

Parses GDELT's own `V2Themes`, `V2Locations`, and `V2Tone` sub-fields into the flat columns the notebook expects, and reports how many rows were dropped and why (missing date, no themes, no usable location).

### 4. Scrape full article text (Stage 3)

Per the project methodology, topic modelling runs on full article text, not just GDELT's theme tags. Test on a small sample first:

```bash
python Scrape_articles.py --input gdelt_cleaned_dataset.csv --out test_articles.jsonl --manifest test_scrape_manifest.csv --limit 20
```

**Before running at scale, edit `USER_AGENT` near the top of `Scrape_articles.py`** to include your real contact email — this is standard scraping etiquette and lets site owners reach you rather than just blocking you.

Then run the full set:

```bash
python Scrape_articles.py --input gdelt_cleaned_dataset.csv --out article_text.jsonl --manifest scrape_manifest.csv
```

This script:
- Checks each domain's `robots.txt` before fetching, and skips (logs as `disallowed`) anything the site doesn't permit
- Uses a real, identifying `User-Agent` and a 1-second delay between requests
- Filters out pages where fewer than 50 words of text were extracted (usually paywalls or non-article pages), logging the reason
- Supports `--resume`, same as `Extraction.py`

**Not every article will scrape successfully** — paywalls, dead links, and anti-bot measures are normal and expected. Check `scrape_manifest.csv` for a full breakdown of `ok`/`disallowed`/`error` counts and reasons; report this in your methodology as a known limitation, not something to hide. The notebook automatically excludes events with no usable text from topic modelling only — they remain in the dataset for the spatial and regression analysis.

### 5. Run the full analysis (Stage 4)

1. Keep `gdelt_cleaned_dataset.csv` and `article_text.jsonl` in the same folder as the notebook (their default paths already point there).
2. Open `gdelt_misinformation_analysis.ipynb` in Jupyter Notebook / Jupyter Lab.
3. Run all cells top to bottom (**Cell → Run All**, or step through with Shift+Enter).
4. Results (plots, tables, model summaries) render inline under each cell. Cleaned data and key result tables are also written to `outputs/` by the final cell.

## Pipeline Overview

1. **Extract (`Extraction.py`)** — download real GKG v2.1 files directly from `data.gdeltproject.org`, one per day at a fixed sampled time across the chosen window; filter to rows matching documented misinformation-related theme codes; save raw matches untouched plus a full manifest of every source file queried. Handles GKG's occasionally oversized fields and supports `--resume` for interrupted runs.
2. **Clean (`Clean_dataset.py`)** — parse GDELT's own `V2Themes`, `V2Locations`, `V2Tone` sub-fields into flat analysis columns; drop and log rows with no usable date/theme/location; deduplicate on `Record_ID`.
3. **Scrape (`Scrape_articles.py`)** — fetch full article text for each event's `Source_URL`, respecting `robots.txt`; log every attempt (`ok`/`disallowed`/`error`) to a manifest.
4. **Theme identification (full-text LDA)** — preprocess scraped article text with spaCy (tokenise, remove stopwords, lemmatise), build a filtered document-term matrix, fit LDA across a range of topic counts, select the best by coherence (`c_v`), assign each event a dominant topic. Events with no scraped text are excluded from this step only.
5. **Exploratory analysis** — theme frequency (from GKG theme tags), dominant narratives by country, tone distributions, tone by location resolution and by country, tone by LDA topic.
6. **Spatial analysis** — map events, run DBSCAN clustering (with a k-distance plot to guide `eps`), compute Moran's I on country-level event counts and tone, generate a KDE hotspot map.
7. **Contextual variables** — derive `Political_Context`, `Economic_Context`, `Health_Context` flags from verified real GKG theme tags, aggregate to a Country × Month panel.
8. **Regression modelling** — fit Poisson regression, test for overdispersion, fit Negative Binomial regression (with MLE-estimated dispersion parameter), report Incidence Rate Ratios (IRRs).
9. **Final outputs** — summary tables, an IRR effect plot, and exported CSVs for reporting.

## Troubleshooting

| Problem | Fix |
|---|---|
| `ModuleNotFoundError` for any package | `pip install -r requirements.txt` (make sure you're in the same Python environment your `python` command actually uses) |
| `PermissionError` writing a CSV | Close any program that has the file open (Excel, another terminal); try a different output filename to confirm |
| Long extraction run stopped partway | Re-run the same command with `--resume` added |
| Raw file and manifest show different date ranges | Run `Repair_manifest.py` before using `--resume` |
| `pip install -r requirements.txt` fails with a long-path / `.js.map` error on Windows | That's Windows Long Path support, not your project — either enable it (see the error's linked docs) or install packages individually, skipping `jupyter` if you already have it via Anaconda |
| Extraction script not found | Confirm you're in the folder containing the `.py` files (`dir *.py` on Windows) and that the file wasn't accidentally deleted |
| `_csv.Error: field larger than field limit (131072)` | Some GKG rows on high news-volume days have an oversized field (usually `AllNames`, `GCAM`, or `Extras`). `Extraction.py`, `Clean_dataset.py`, and `Repair_manifest.py` all raise Python's CSV field-size limit to handle this — make sure you're running the current version of whichever script raised the error |
| Most/all articles show `status=error` in `scrape_manifest.csv` | Check `reason` in the manifest — common causes are paywalls, dead links, or sites that block automated requests entirely. Some failure rate is normal and expected; report it in your methodology rather than treating it as a bug |
| Most/all articles show `status=disallowed` | The domains you're scraping have restrictive `robots.txt` files. This is a legitimate result, not something to bypass — reflect it honestly in your write-up as a data-collection limitation |
| `OSError: [E050] Can't find model 'en_core_web_sm'` | Run `python -m spacy download en_core_web_sm` (the notebook's install cell does this automatically) |

## Notes for Interpretation

- **Geography:** report DBSCAN cluster count/locations, whether Moran's I indicates significant spatial clustering (vs. randomness), and where KDE hotspots fall.
- **Context:** report which of the Political/Economic/Health context proportions have an IRR significantly different from 1, and in which direction.
- `EPS`/`MIN_SAMPLES` (DBSCAN) and `NUM_TOPICS` (LDA) are analytic choices made in the notebook — revisit and justify them for your own write-up rather than leaving them at their defaults.
